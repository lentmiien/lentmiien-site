The tool `/admin/request-counter` is a proof of concept of a more extensive management tool that I want to create.

So, using the request counter as base, I want to start working on a more advanced tool, with more functionality.

First of all, this is a tool for logging usage from 1 unique device, and return a JSON object that tell the device what it should do next, for example in the proof of concept version I only returned "OK" or "NG" to restrict usage of the device to max 60 minutes per 90 minutes rolling window, which should remain the 'core' of the more advanced tool, but I want to add more nuance to the tool, and more management options.

The input will remain the same "once per minute, currently what 'package' (app) is running", but I want to add to the internal server logic and make the responses contain more information for the device to act on.

First major change: I want to categorize 'packages' into 3 categories ("learning", "management" and "entertainment"; "entertainment" will roughly map to the existing tool, and be the default category for 'unknown' package values) *So I need a database mapping 'known' packages to my categories.

The next change: "entertainment" should require completion of 30 minutes of "learning", before activating for the day, and the first 30 minutes of learning should be "free" (not affect the 60 minutes of usage per 90 minutes rolling window), after which learning and entertainment join together for max 60 minutes per 90 minutes rolling window; the "management" category is used for app used by adults, and should be ignored from the counter.

I want to add some sort of 'points'/'stars' system, the device is being used by a child, and I want an adult management dashboard where adults can monitor what the child is doing, and reward the child for doing good things (these rewards aren't a core part of the app, but I likely want to store rewards with comments, to avoid duplicate rewards etc. and store reward suggestions in a database, for example "clean room", to get quick select for reoccuring types of rewards).

We could probably add some 'points'/'stars' to real life reward exchange system in the future, but this require more consideration to make a practical and balanced system.

Also, I would like to create some learning system within this app as well, for example a math game, that can save more detailed progress for adult review and more clear 'points'/'stars' reward, for a per question reward, instead of simply "using a learning app for xx minutes", where I don't know what the child did for the xx minutes. But this is not something I want to include initially, as I need more time to design this system, so just make sure the what we do now is future proof for a system like this.

Initially, I want to keep the current request counter, until I'm ready for the switch; I want the new tool to have a new random URL and use the same request input format for an easy transition, but use a new JSON return format with more details for the device to know what it need to do next; I also want a more rich admin dashboard for managing usage across the categories and restrictions for each category, and giving rewards, etc.

I currently expose the current request stats on a "my wife" view, in Japanese, and I plan to add this tool there when done, with the add reward system, but this specific implementation can come a bit later, just make sure that any text labels support both English and Japanese, so the addition on my wifes page is simpler when done.

Now, create the base system, with a new random URL, and a new admin page, then we'll go from there.
