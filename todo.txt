I'd like to add a new learning/quiz feature to the app that replaces the current standalone chemistry prototype (`public/html/mini-chem-quest.html`) with a proper integrated, database-backed system. This should not be treated as a one-off chemistry page moved into Pug; it should become a reusable learning tool inside the existing Node/Express app, built in a way that fits the codebase and can support many subjects over time. Chemistry will be the first topic we use to seed it, but the feature itself needs to be generic and content-driven from the start. Please add a new route/controller setup for this tool and wire it into the app in whatever way best matches the current architecture, with the learner-facing page rendered from a Pug template and its CSS and JavaScript moved into normal separate asset files.

The learner-facing experience should still feel like the current chemistry mini-game rather than a traditional quiz form. It needs to be very visual, playful, highly interactive, and suitable for young children. The UI should be touch-friendly and mobile-friendly, with large tap targets, simple prompts, lots of graphics, and minimal text. The goal is for it to feel fun and encouraging, not academic or text-heavy. I want to preserve the rewarding parts of the prototype: immediate feedback, stars or points, completion moments, stickers or badges, and a sense of progress. If a child opens the tool, it should feel like they are entering a friendly learning space, not filling out a worksheet. The frontend can use simple SVG-style graphics and lightweight animations; we do not need heavy custom art to make the experience feel polished.

The content structure should be based around topics and subtopics. Chemistry is just one topic, and within chemistry we should be able to have subtopics like Atoms, Molecules, and States of Matter, mirroring the prototype. The same structure should also support completely different topics later, such as colors, animals, space, numbers, or anything else. A topic should be a broad category, and a subtopic should be the unit that the learner actually enters and completes. Each subtopic should contain a sequence of learning items. To preserve the feel of the current prototype, those items should not have to be only scored quiz questions; the system should also support non-scored interactive activities or mini-scenes that introduce a concept in a playful way before or between scored questions. In other words, the experience should allow a mix of short activity blocks and actual questions within the same subtopic flow.

All of the content should be stored in the database rather than hardcoded in the frontend. I want to be able to dynamically add topics, subtopics, and questions through the app. The important thing is that the system should be template-based rather than fully custom per question. Each item should use one of a set of predetermined question or activity templates, with its specific content/config coming from the database. For the initial version, the templates should cover the kinds of interactions already used in the chemistry prototype, such as single-choice or image-choice questions, counting-style interactions, simple slot-building or arrangement interactions, and state-change controls like “heat until it becomes gas.” There should also be a non-scored interactive activity template so we can keep the playful intro scenes. The exact internal naming is up to you, but the overall model should make it easy to add more template types later without reworking the whole system.

I want the question/item system to be designed so that adding new content does not require code changes, as long as that content uses an existing template. Code changes should only be necessary when we introduce an entirely new template or new kind of interactive behavior. To make that work well, the app should have a clear separation between content and behavior: the database stores the prompt, options, media, correct answer config, layout config, reward data, and theme overrides, while the application code defines how each template is rendered and how it is evaluated. On the frontend, the JavaScript should render the item based on the template type and its config. On the backend, the answer should be validated and evaluated based on the same template type and stored data. I do not want correctness and progress to depend only on client-side logic; the frontend can handle immediate UI feedback and interactivity, but the server should remain the source of truth for whether an answer was correct and what progress changes should be saved.

The learner route should load the relevant topic or subtopic, the learner’s saved progress, and enough data to initialize the interface cleanly. Whether you render initial content server-side and then hydrate, or render the shell and let the frontend fetch data immediately, is up to you and the current app patterns. The key thing is that when a user opens the tool, they should either land on a topic or subtopic overview or be taken directly into a subtopic, and the page should know their current progress and present the right state. There should be some form of user-facing route for opening the tool, a server endpoint for submitting answers and updating progress, and an admin area for managing the content. The route naming and structure should follow the conventions already used in the app. Deep-linking directly into a topic or subtopic should be supported if it makes sense in the current routing setup.

Progress needs to be persisted per user in the database. The current prototype stores progress in session memory, but for the integrated version I want actual saved progress tied to the user account. At minimum, the system should remember what subtopics a user has completed, how many stars or points they have earned, what stickers or rewards they have unlocked, and enough per-item progress to resume a lesson cleanly. If it makes sense with the architecture, keeping answer attempt history or lightweight analytics data would also be useful.

The admin side should function like a lightweight CMS for this learning tool. I need to be able to add, edit, organize, and publish topics, subtopics, and items/questions from within the app. That should include setting titles, descriptions, ordering, slugs or identifiers if needed, published/draft state, and any associated reward or theme data. When creating an item, I should be able to choose one of the supported templates and then enter the template-specific content through an appropriate UI. For example, a single-choice question should show fields for prompt, options, correct answer, and optional graphics; a builder or arrangement interaction should show fields for pieces, slots, and target arrangement; a visual activity block should show the settings relevant to that activity. The admin should also be able to assign or upload graphics/assets where needed, set theme overrides, and ideally preview the content in a learner-like view before publishing. I do not mind if the admin experience is a little more utilitarian than the learner experience, but it should still be structured and safe enough that we are not hand-editing raw database records to create content.

Please also make sure there is some clear concept of published versus draft content so we can work on topics without exposing incomplete material to users. Ordering and visibility matter here, because topics, subtopics, and items need to appear in a predictable sequence. Stable IDs and proper content references are important as well, since progress will be tied to individual content records and we do not want user progress to become fragile every time content is updated. Chemistry should exist as the initial working topic and should reproduce the spirit of the standalone prototype, but it should be just data plus the standard templates rather than a hardcoded exception in the code.

From a data perspective, I expect the system to have persistent models or equivalent structures for topics, subtopics, learning items/questions, template config, theme/asset references, and per-user progress. If it fits the current stack, I’d also like there to be a place for answer attempt history and reward unlocks. I am not prescribing a specific schema here because you have the codebase and can align it with the current conventions, but the feature should be implemented in a way that makes these concepts first-class and not buried in ad hoc JSON with no structure. The same goes for routes, controllers, services, and frontend organization: please shape those around the app’s existing architecture, but make sure the final system clearly supports the content model and learner experience described above.

We should be able to say this feature is successfully created when a logged-in user can open the new learning tool in the app, see a polished child-friendly interface, navigate into a topic and subtopic, play through interactive learning items and questions, receive immediate visual feedback, earn stars and stickers, and return later with their progress still intact in the database. We should also be able to log into the admin side, create a new topic and subtopic, add questions or activities using the supported templates, publish that content, and see it appear in the learner UI without needing new frontend code. Topic-specific theming should work when provided, and the default theme should be used automatically when it is not.

I’m intentionally not trying to dictate the exact implementation approach, because you have the full codebase and can fit this into the current app structure much better than a rigid spec would. What matters most is that the final feature is reusable, content-driven, pleasant for children to use, manageable through the app, and strong enough that chemistry is just the first example rather than the only subject it can support.
